## 二、技术栈选型

### 前端

| 技术 | 选型 | 原因 |
|------|------|------|
| 框架 | **React + Next.js 14** | SSR/SSG，SEO友好，性能好 |
| UI 组件库 | **shadcn/ui + Tailwind CSS** | 现代化、可定制、轻量 |
| 图表 | **Recharts / ECharts** | 概率分布图、趋势图 |
| 状态管理 | **Zustand** | 轻量，比 Redux 简洁 |
| 类型系统 | **TypeScript** | 全栈类型安全 |

### 后端

| 技术 | 选型 | 原因 |
|------|------|------|
| 框架 | **FastAPI (Python 3.11+)** | 异步高性能，自动生成 OpenAPI 文档 |
| ORM | **SQLAlchemy 2.0 + Alembic** | 成熟稳定，迁移管理 |
| 任务队列 | **Celery + Redis** | 异步模型训练、数据同步 |
| 认证 | **JWT + OAuth2** | 无状态认证 |

### 数据存储

| 技术 | 用途 |
|------|------|
| **PostgreSQL** | 主数据库：比赛数据、球队、联赛、用户 |
| **Redis** | 缓存、会话、实时赔率、任务队列 |
| **MinIO / S3** | 模型文件存储（.joblib / .pkl） |
| **ClickHouse**（可选） | 海量历史数据聚合分析 |

### AI/ML 引擎

| 技术 | 用途 |
|------|------|
| **scikit-learn** | 基础模型、Pipeline、Stacking |
| **LightGBM + XGBoost + CatBoost** | 梯度提升集成模型 |
| **NumPy / SciPy** | 泊松分布、蒙特卡洛模拟 |
| **Optuna** | 超参数贝叶斯搜索 |
| **joblib** | 模型序列化 |
| **MLflow** | 实验追踪、模型版本管理 |

### 基础设施

| 技术 | 用途 |
|------|------|
| **Docker + Docker Compose** | 本地开发环境 |
| **GitHub Actions** | CI/CD |
| **Nginx** | 反向代理 + 静态资源 |
| **Sentry** | 错误监控 |
| **Grafana + Prometheus** | 系统监控 |

## 四、量化预测引擎 — 五步决策管道

核心分析方法论：**不依赖单一模型，而是模拟职业分析师的多层次验证流程**。

### 4.1 五步决策管道总览

### 4.2 Step 1: 基本面 + 盘赔组合 → 判断比赛类型

**目标**: 通过 TSI (Team Strength Index) 差距与亚盘深度的交叉，将比赛归入 6 种类型。

| 比赛类型 | TSI 差距 | 亚盘范围 | 特征 | 策略含义 |
|----------|----------|----------|------|----------|
| **强队碾压型** | >15 | -3.0 ~ -1.5 | 实力绝对碾压，深盘支撑 | 正常分析即可 |
| **中等优势型** | 8~15 | -2.0 ~ -0.75 | 一方占优但不绝对 | 关注机构共识度 |
| **实力接近型** | 0~8 | -0.75 ~ 0.25 | 五五开，平手/平半 | 指数验证权重↑ |
| **冷门风险型** | >10 | -0.25 ~ 0.25 | 强队浅盘，盘赔背离 | 🚨 大幅降仓 |
| **德比/特殊** | — | — | 德比战意主导 | 基本面权重↓ |
| **价值陷阱型** | <8 | <-1.0 | 低赔深盘但实力不足 | ⛔ 建议回避 |

**核心实现**: `engine/classifier/match_classifier.py`
- 规则矩阵 6 条分类规则
- 冷门风险信号检测（强队浅盘、赔率漂移、关键伤缺）
- 价值陷阱检测（低赔 + 深盘 + 实力不匹配）
- 德比自动识别

### 4.3 Step 2: 机构统一思路 → 确定比赛方向

**目标**: 分析 10+ 主流博彩机构的赔率共识程度，判断市场方向。

| 共识级别 | 赔率 CV | 含义 | 权重 |
|----------|---------|------|------|
| **高度一致** | <3% | 机构方向明确 | 35% |
| **中度共识** | 3~7% | 主流趋同，少数分歧 | 35% |
| **弱共识** | 7~12% | 分歧较大 | 25% |
| **明显分歧** | >12% | 方向不确定 | 10% |

**核心实现**: `engine/consensus/consensus_analyzer.py`
- 加权均值（Pinnacle/Bet365 权重最高）
- 变异系数 (CV) 量化离散度
- 隐含概率 → 市场方向判定
- 赔率走势检测（steaming vs drifting）

### 4.4 Step 3: 指数辅助 → 核实方向

**目标**: 用三种量化指数验证 Step 2 的方向，交叉确认或发现矛盾。

| 指数 | 作用 | 矛盾信号 |
|------|------|----------|
| **凯利指数** | 衡量投注价值，正凯利 = 价值方向 | 凯利方向 ≠ 机构方向 |
| **成交量热度** | 市场资金流向，>70%单边 = 过热危险 | 过热方向 = 反向信号 |
| **赔率稳定性** | 赔率波动程度，不稳则方向可信度↓ | 稳定度 < 0.6 |

**核心实现**: `engine/validator/index_validator.py`
- 凯利值计算（去水后公平概率 × 赔率）
- 过热检测（单边 > 70% 视为过热）
- 平局信号检测（平赔异常偏低）
- 综合支持度评分 (-1 ~ 1)

### 4.5 Step 4: 市场定价检验

**目标**: 将模型估计的「真实概率」与市场赔率的「隐含概率」对比，寻找定价偏差。

```
EV (期望价值) = 模型估计概率 × 市场赔率 - 1

  EV > +5%  → 被低估（价值洼地）
  -3% < EV < +5%  → 定价合理
  EV < -3%  → 被高估（无价值）
```

**核心实现**: `engine/pricing/pricing_checker.py`
- 公平赔率 vs 市场赔率偏差计算
- EV 三向计算（主/平/客）
- 水位结构分析（返还率 + 偏差分布）

### 4.6 Step 5: 综合判断

**目标**: 加权融合前四步的结论，输出最终预测。

| 信号来源 | 权重 | 说明 |
|----------|------|------|
| Step 2 机构共识 | 35% | 最直接的市场方向信号 |
| Step 3 指数验证 | 25% | 量化确认/否定 |
| Step 4 市场定价 | 25% | 价值判断 |
| Step 1 比赛类型 | 15% | 情境参考 + 风险权重 |

**融合逻辑**:
1. 按方向聚合各步骤信号 → 归一化概率
2. 概率差 > 10% → HIGH_CONFIDENCE
3. 概率差 > 5% → MODERATE
4. 价值陷阱触发 → SKIP（自动回避）

**核心实现**: `engine/orchestrator/pipeline.py` — `run_full_pipeline()`
- 五步编排
- 加权融合
- 风险集成
- Markdown 可读报告自动生成

### 4.7 TSI (Team Strength Index) 评分公式

| 因子 | 权重 | 数据来源 |
|------|------|----------|
| ELO 评分（归一化） | 25% | 自维护 ELO 系统 |
| xG For（近10场均值） | 15% | FootyStats / Opta |
| xG Against（近10场均值） | 15% | 同上 |
| 近期战绩得分（近5场） | 15% | 比赛历史数据库 |
| FIFA / 联赛排名 | 10% | 官方排名 API |
| 阵容完整度 | 10% | 伤病/停赛数据 |
| 主客场因子 | 10% | 历史统计 |

### 4.8 ML 辅助模型（按需增强）

在五步决策管道之外，保留 ML 模型轨道用于特定场景增强：

| 场景 | 模型 | 用途 |
|------|------|------|
| Step 4 概率输入 | Stacking(LGB+XGB+Cat) | 提供"模型估计概率" |
| 比分预测 | 泊松 + 蒙特卡洛 | 比分分布 |
| 大小球 | StackingClassifier × 3 | >1.5 / >2.5 / >3.5 |
| BTTS | StackingClassifier | 双方进球

## 六、API 设计

```
Base URL: https://api.yourdomain.com/v1

# 预测
POST   /predict                    # 单场比赛预测
POST   /predict/batch              # 批量预测（一轮比赛）
GET    /predict/history/{date}     # 历史预测结果

# 数据查询
GET    /leagues                    # 联赛列表
GET    /leagues/{id}/standings     # 积分榜
GET    /leagues/{id}/teams         # 球队列表
GET    /matches?date=&league=     # 赛程查询
GET    /matches/{id}               # 比赛详情
GET    /teams/{id}/profile         # 球队画像
GET    /teams/{id}/h2h/{opponent}  # 历史交锋

# 赔率
GET    /odds/{match_id}            # 实时赔率

# 回测
POST   /backtest/run               # 发起回测任务
GET    /backtest/{id}/status       # 回测状态
GET    /backtest/{id}/report       # 回测报告

# 模型
GET    /models                     # 模型列表
GET    /models/{id}/metrics        # 模型评估指标
```

## 十一、关键技术决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 预测引擎语言 | Python | ML 生态最完善，scikit-learn/LightGBM 等 |
| 前端框架 | Next.js | SSR SEO友好，React 生态成熟 |
| 数据库 | PostgreSQL | 支持复杂分析查询，扩展性好 |
| 模型热更新 | joblib + Redis Pub/Sub | 无需重启服务即可加载新模型 |
| 回测 | 独立 Celery 任务 | 避免阻塞 API，可扩展 |
| 实时性 | 赔率 5min 轮询 + 赛后 30min 更新 | 平衡成本与时效 |

## 十二、快速启动

```bash
# 1. 克隆项目
git clone <your-repo>
cd football-quant-prediction

# 2. 配置环境变量
cp .env.example .env
# 填入 API-Football Key、DB密码等

# 3. 启动所有服务
docker-compose up -d

# 4. 初始化数据库
docker-compose exec backend alembic upgrade head

# 5. 导入基础数据（英超为例）
docker-compose exec backend python -m data.etl.bootstrap --league epl

# 6. 训练初始模型
docker-compose exec backend python -m engine.ml.train_football

# 7. 访问
# 前端: http://localhost:3000
# API文档: http://localhost:8000/docs
```

> **核心理念**: 这个框架采用「统计模型 + ML模型」双轨制，统计模型保证可解释性，ML模型追求准确率，通过贝叶斯模型平均融合两者优势。参考了 FiveThirtyEight 的 SPI、Opta 的数据维度以及业界生产级实践。
