# FootballQuant 前端分析工作台设计

日期：2026-07-03

## 目标

重构前端首屏为“专业分析工作台”，把项目核心能力集中呈现：

- 手动选择比赛，支持搜索、筛选、48 小时/今日/价值/关注视图。
- 队伍名中文优先展示，英文/API-Football 名称作为副标题保留。
- 胜平负、亚盘、大小球、角球四大市场用统一概率卡展示。
- 点击查看分析报告时，展示结构化报告模板，而不是只显示原始文本。
- 保留专业终端式信息密度，同时保持可读性和操作路径清晰。

## 信息架构

首屏采用三栏工作台：

- 左侧比赛列表：中文队名、英文名、联赛、开赛时间、赔率覆盖、风险/价值标签。
- 中间分析区：选中比赛标题、核心 KPI、四大市场卡片、11 维分析、市场排序、复盘摘要。
- 右侧操作区：赔率快照、风险提示、Telegram 推送、关注、人工复核、任务日志。

## 中文队名策略

后端返回 `home_team_zh` / `away_team_zh`，前端不硬编码中文队名。

解析优先级：

1. 数据库 `team_aliases` 中与 API-Football 名称绑定的中文别名。
2. 本地 `team_aliases.seed.json` 中的种子中文别名。
3. API-Football team id 后续绑定的人工修正结果。
4. 未命中时回退英文名，并在 UI 上保留英文副标题。

## 市场卡片

每场比赛返回 `market_cards` 数组，前端按类型渲染：

- `1x2`：主胜、平局、客胜。
- `asian_handicap`：主队盘口、客队盘口、走水/保护提示。
- `over_under`：大球、小球、节奏倾向。
- `corners`：大角、小角、边路/角球倾向。

每张卡包含：

- `title`
- `subtitle`
- `outcomes[]`，包含 label、probability、tone
- `metrics[]`，包含 label、value
- `action`

数据不足时仍显示卡片，但标记“数据不足/待采集”，避免页面结构跳动。

## 分析报告模板

报告分为结构化区块：

- 比赛信息
- 核心建议
- 四大市场拆解
- 11 维分析摘要
- 风控提示
- 操作建议
- 原始引擎报告

前端优先展示结构化模板；如果后端只有 `report_text`，则作为原始引擎报告放在底部。

## 实现边界

本轮实现聚焦展示层和现有数据聚合：

- 不新增数据库表。
- 不改变 Celery 自动分析逻辑。
- 不强行创造真实角球赔率；若当前数据不足，展示“待采集/分析信号”卡片。
- 不接入真实 Telegram 推送按钮行为，只保留前端操作入口。

- 后端 Python 语法检查通过。

- 首页在无数据时应显示空状态，不报错。
- 有比赛数据时能手动选择比赛并展示市场卡片和报告模板。
